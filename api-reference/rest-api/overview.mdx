## How It Works

The REST API exposes your MainWP Dashboard functionality through standard HTTP endpoints. External tools send authenticated requests to perform actions like listing sites, triggering updates, or managing clients. Responses return structured JSON data that can be parsed by any programming language or automation platform.

---

## Prerequisites

| Requirement | Details |
|-------------|---------|
| MainWP Dashboard | Version 5.2 or later |
| Permalinks | Pretty permalinks enabled (not "Plain") |
| API Key | Generated in Dashboard > API Access > API Keys |
| Network Access | Dashboard must be publicly accessible |

<Note>
Your MainWP Dashboard must be accessible via the internet. Locally hosted Dashboards cannot receive external API requests.
</Note>

---

## Base URL

All API requests target your Dashboard's WordPress REST API namespace:

```
https://your-dashboard.com/wp-json/mainwp/v2/
```

Replace `your-dashboard.com` with your actual Dashboard domain.

### Create an API Key

<Steps>
<Step title="Configure permalinks">
Go to **WP Admin > Settings > Permalinks** and choose any option except **Plain**.

![Permalinks settings](/images/add-ons/c87f79edc47e.jpg)
</Step>
<Step title="Open Add API Keys">
Go to **Dashboard > API Access > API Keys > Add API Keys**.
</Step>
<Step title="Generate the key">
Enter a descriptive name and click **Generate**.
</Step>
<Step title="Copy the Bearer token">
Copy the generated API key immediately and store it securely.

![Add API Key form showing the generated Bearer token](/images/dashboard/dashboard-overview-4f2818.png)

<Warning>
The API key value is shown only once.
</Warning>
</Step>
<Step title="Assign permissions">
Select the minimum required permissions for your integration.
</Step>
<Step title="Optional: enable legacy v1 credentials">
Click **Show Legacy API Credentials** to reveal the Consumer Key and Consumer Secret, then enable **MainWP REST API v1 Compatibility** only if needed.

![Legacy API credentials and v1 compatibility toggle](/images/dashboard/dashboard-overview-75e153.gif)
</Step>
<Step title="Save settings">
Click **Save Settings** to activate the key.
</Step>
</Steps>

API access remains active as long as at least one API key is enabled.

### API Key Permissions

| Permission | Methods | Description |
|------------|---------|-------------|
| **Read** | GET | Retrieve data only |
| **Write** | POST, PUT | Create resources or trigger actions |
| **Delete** | DELETE | Remove resources |

<Note>
Keys created before MainWP 4.5 had all permissions enabled by default. Review older keys and reduce scope where possible.
</Note>

![API Key permissions table showing Read, Write, and Delete options](/images/dashboard/dashboard-overview-cab613.png)

### Use API Keys in Requests

Include the key as a Bearer token in the `Authorization` header:

```bash
curl -X GET "https://your-dashboard.com/wp-json/mainwp/v2/sites/basic" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### Disable API Keys

To temporarily block access without deleting credentials, go to **Dashboard > API Access > API Keys > API Keys** and disable the key.

![Manage API Keys page showing enabled keys with disable toggle](/images/dashboard/dashboard-overview-0479f2.png)

### Delete API Keys

Go to **Dashboard > API Access > API Keys > API Keys** and delete individual keys, or use bulk actions to delete multiple keys.

![Manage API Keys page with Delete button and Bulk actions dropdown](/images/dashboard/dashboard-overview-4a7ea2.png)

### Legacy API (v1)

Legacy v1 endpoints use Consumer Key and Consumer Secret credentials.

```bash
curl -X GET "https://your-dashboard.com/wp-json/mainwp/v1/sites" \
  -u "CONSUMER_KEY:CONSUMER_SECRET"
```

<Note>
Use v1 only for backward compatibility. New integrations should use v2 with Bearer tokens.
</Note>

### Authentication Troubleshooting

| Error | Cause | Fix |
|-------|-------|-----|
| `401 Unauthorized` | Missing or invalid token | Re-copy the key and verify `Authorization: Bearer ...` format |
| `403 Forbidden` | Permission mismatch | Update key permissions (Read/Write/Delete) |
| `rest_forbidden` | Endpoint requires a different method/permission | Confirm endpoint and HTTP method match key scope |

## Request Format

The API accepts parameters in different ways depending on the HTTP method:

- **GET requests**: Pass parameters as URL query strings
- **POST/PUT requests**: Send parameters as JSON in the request body
- **DELETE requests**: Usually require only the resource identifier in the URL

Include the `Content-Type: application/json` header when sending JSON bodies:

```bash
curl -X PUT \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name": "My Site", "url": "https://example.com"}' \
  "https://your-dashboard.com/wp-json/mainwp/v2/sites/add"
```

---

## Response Format

### Success

All responses return JSON with the requested data:

```json
{
  "id": 1,
  "url": "https://example.com",
  "name": "My Site",
  "status": "connected"
}
```

### Errors

Error responses include a code and message:

```json
{
  "code": "rest_forbidden",
  "message": "Sorry, you are not allowed to do that.",
  "data": {
    "status": 403
  }
}
```

## Pagination

List endpoints support pagination through query parameters:

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `page` | integer | 1 | Current page number |
| `per_page` | integer | 10 | Items per page |

Response headers include pagination metadata:

| Header | Description |
|--------|-------------|
| `X-WP-Total` | Total number of items |
| `X-WP-TotalPages` | Total number of pages |

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" \
  "https://your-dashboard.com/wp-json/mainwp/v2/sites/?page=2&per_page=25"
```

## Batch Operations

When triggering batch updates, the API responds immediately with confirmation rather than waiting for completion:

```json
{
  "success": 1,
  "message": "Batch updates all started successfully.",
  "last_time_start_batch_updates": "2024-09-03T16:57:29"
}
```

<Warning>
Sync your Dashboard after allowing time for batch operations to complete to see updated results.
</Warning>

For update-specific behavior, see [Updates Endpoints](/api-reference/rest-api/updates).

## Error Reference

### HTTP Status Codes

| Code | Description |
|------|-------------|
| 200 | Request succeeded |
| 201 | Resource created |
| 400 | Bad request - invalid parameters |
| 401 | Authentication required or invalid |
| 403 | Permission denied |
| 404 | Resource or endpoint not found |
| 405 | Method not allowed |
| 5XX | Server error |

### MainWP-Specific Errors

| Code | HTTP | Description |
|------|------|-------------|
| `mainwp_site_not_found` | 404 | Site ID or domain not found |
| `mainwp_client_not_found` | 404 | Client ID or email not found |
| `mainwp_site_offline` | 503 | Site unreachable |
| `mainwp_invalid_parameters` | 400 | Required parameter missing or invalid |
| `rest_forbidden` | 403 | API key lacks required permission |

## Feature-Gated Endpoints

Some endpoints require optional MainWP extensions:

| Extension | Required For |
|-----------|--------------|
| Cost Tracker | `/costs` endpoints |
| SSL Monitor | `/ssl-monitor` endpoints |
| Pro Reports | `/pro-reports` endpoints |

If the required extension is inactive, these endpoints return a 404 or feature-specific error.
